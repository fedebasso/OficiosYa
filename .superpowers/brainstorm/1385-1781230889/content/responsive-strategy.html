¿Cómo implementamos el sistema responsive?

El brief pide mobile-first real con Tailwind. Hay 3 formas de atacarlo — cada una tiene un trade-off importante.

A

CSS Custom Properties + clamp()

Definir variables CSS con clamp(min, preferred, max) para tipografía, spacing y tamaños. Fluido entre breakpoints sin saltos. Un solo archivo de tokens (tokens.css) controla todo.

Pros

  • Sin saltos bruscos entre viewports
  • Escalado matemáticamente consistente
  • Un solo lugar para ajustar todo

Contras

  • Más difícil de depurar visualmente
  • Requiere reemplazar estilos inline actuales
B

Tailwind Breakpoints (sm/md/lg) — mobile-first

Usar los breakpoints de Tailwind correctamente: base = 320px, sm: = 375px, md: = 430px, lg: = 768px. Migrar los estilos inline clave a clases Tailwind.

Pros

  • Familiar, predecible, fácil de leer
  • Encaja con el stack actual
  • Fácil de mantener por otros devs

Contras

  • Saltos entre breakpoints (no fluido)
  • Mezcla con estilos inline existentes
C

Híbrido: tokens CSS + Tailwind para layout

CSS custom properties para tipografía y spacing (fluido con clamp), Tailwind para layout y grid (breakpoints). Lo mejor de los dos mundos — estilos inline se migran a variables, layout usa clases.

Pros

  • Tipografía y spacing fluidos
  • Layout predecible con Tailwind
  • Profesional y escalable

Contras

  • Más setup inicial
  • Dos sistemas a entender

💡 Mi recomendación: C (Híbrido) — es lo que usan Uber y Airbnb internamente. Fluido donde importa, predecible donde importa.
